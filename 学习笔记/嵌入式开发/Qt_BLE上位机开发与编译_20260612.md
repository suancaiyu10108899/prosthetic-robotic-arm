# Qt 6.11 BLE 上位机开发与编译

> **日期**：2026-06-12
> **主题**：用 Qt 6 + CMake 构建 BLE 桌面调试工具
> **项目**：arm-ble-gui（假肢机械臂上位机）

---

## 一、为什么用 Qt 而不是 Flutter/Electron？

| 方案 | BLE 支持 | 桌面部署 | 与你技能栈一致性 |
|------|:---:|:---:|:---:|
| Qt 6 C++ | ✅ QBluetooth (原生 WinRT) | ✅ 单 exe + 少量 DLL | ⭐⭐⭐⭐⭐ (MindPath 同款) |
| Flutter | ⚠️ 需要 Method Channel 桥接 | ⚠️ Windows 桌面不是一等公民 | ⭐⭐ |
| Electron | ⚠️ 需要 noble/bleno 原生模块 | ❌ 几百 MB 安装包 | ⭐ |

选择了 Qt — 与 MindPath 相同的 CMake + C++17 + MSVC 技术栈。

## 二、项目骨架 — 四层架构

延续 MindPath 的分层惯例：

```
src/
├── ble/          ← BLE 层：Qt6::Bluetooth，不依赖 Widgets
│   ├── BleDeviceInfo      纯数据结构
│   ├── BleScanner         扫描封装
│   ├── BleConnection      连接/服务发现/收发
│   └── NusService         NUS UUID 常量
├── ui/           ← UI 层：Qt6::Widgets
│   ├── MainWindow         编排者（依赖注入组装）
│   ├── DeviceScanPanel    设备扫描面板
│   ├── PresetPanel        预设指令按钮
│   ├── SendPanel          发送面板
│   └── LogPanel           接收日志
└── app/          ← 入口：依赖组装
    └── main.cpp
```

**分层规则**：
- `ble/` 只依赖 `Qt6::Core` + `Qt6::Bluetooth` — 不知道 UI 的存在
- `ui/` 依赖 `ble/` + `Qt6::Widgets` — 把 BLE 信号连接到 UI 控件
- `app/main.cpp` 组装所有依赖 — 只在这里 `new` 对象

## 四、Qt 6.11 BLE API 踩坑记录

### 坑1：createCentral 不再接受 QBluetoothAddress

Qt 6.11 中 `QLowEnergyController::createCentral()` 的签名变了：

```cpp
// ❌ 旧写法（Qt 6.5 及以前）
m_controller = QLowEnergyController::createCentral(
    QBluetoothAddress(address), this);

// ✅ 新写法（Qt 6.11）
QBluetoothDeviceInfo deviceInfo(QBluetoothAddress(address), QString(), 0);
m_controller = QLowEnergyController::createCentral(deviceInfo, this);
```

### 坑2：enum class State 无法通过 connect 传递给 lambda

```cpp
// ❌ Qt 6.11 + MSVC 模板推导失败
connect(m_connection, &BleConnection::stateChanged,  // 签名: void(State)
        this, [this](State s) { ... });              // 不匹配！

// ✅ 把信号参数改为 int，emit 时 cast
signals:
    void stateChanged(int newState);  // 用 int 代替 enum class
```

**原因**：Qt MOC 的 `Q_ENUM(State)` 注册与 MSVC 模板 SFINAE 在 Qt 6.11 下有兼容问题。这是 Qt 自身的 bug，后续版本可能修复。

### 坑3：BleDeviceInfo 需要持有完整的 QBluetoothDeviceInfo

因为 `createCentral` 需要 `QBluetoothDeviceInfo` 对象（不只是 MAC 地址），所以扫描时不能扔掉 Qt 原始对象。

## 五、BLE 连接完整流程

```
[扫描]
BleScanner::startScan()
  → QBluetoothDeviceDiscoveryAgent::start(LowEnergyMethod)
  → 每发现设备 emit deviceDiscovered(BleDeviceInfo)
  → 设备列表显示在 UI

[连接]
用户选择设备 → connectToAddress(mac)
  → QLowEnergyController::createCentral(deviceInfo)
  → connectToDevice()
  → 连接成功 → discoverServices()
  → 发现 NUS (6E400001-...) → createServiceObject()
  → discoverDetails() → 找到 TX/RX 特征
  → writeDescriptor(CCCD, 0100) 启用通知
  → State::Ready → 可以收发

[收]
RX 特征通知 → onCharacteristicChanged → emit textReceived()

[发]
用户输入 → BleConnection::sendText() → writeCharacteristic(TX, data)
```

## 六、构建流程

```powershell
# 每次新终端第一步
. .\scripts\dev-shell.ps1    # 激活 MSVC 环境

cmake --preset win-msvc-debug               # 配置
cmake --build --preset win-msvc-debug        # 编译
cd build\win-msvc-debug
ctest --output-on-failure                    # 测试
.\src\app\arm-ble-gui.exe                   # 运行
```

不需要 Qt Creator — VSCode + CMake Tools 就够了。

## 七、要点总结

1. Qt 6.11 的 Bluetooth API 有两处破坏性变更，需适配
2. 四层架构（app → ui → ble）与 MindPath 一致，依赖严格单向
3. BLE 连接有三层状态（GATT 连接 → 服务发现 → 特征就绪），每一步都可能失败
4. NUS UUID 在 6/11 已通过 nRF Connect 验证，不需要重新抓包
5. `windeployqt` 可以自动打包 Qt DLL 让 exe 独立运行
