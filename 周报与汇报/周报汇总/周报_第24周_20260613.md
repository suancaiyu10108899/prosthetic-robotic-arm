## 三、解决方案 / 思路

- **Qt 6.11 API 破坏性变更**：在 `04_上位机/README.md` 和开发日志中记录，供后续参考
- **手柄协议分析**：未用文档描述，直接从 Arduino 库源码（`codex_pad.cpp`）反推 UUID 和数据格式，比文档更可靠
- **手柄固件架构**：不设虚接口，用 `HandleType` 枚举 + `parseInput()` switch 分派，Arduino 上简单可靠
- **两个手柄差异**：CodexPad 8字节二进制（模拟摇杆），LOOKBON 2字节 ASCII（开关量），统一归一到 `ParsedInput`

## 六、学习收获

1. **BLE 协议栈实践**：从 nRF Connect 验证 → Qt API 封装 → Arduino NimBLE 库，完整走通了 BLE 三层（广播→连接→数据）
2. **Qt 6 跨线程信号/槽**：Qt 6.11 的 `connect` 模板对 `enum class` 信号推导失败，需用 `int` 中转；自定义结构体跨线程传递需 `qRegisterMetaType` + 值传递
3. **从源码反推协议**：当文档缺失时，`git clone` 官方 Arduino 库，从 `codex_pad.cpp` 中找到 Service/Char UUID 和数据帧结构，比等商家文档更快
4. **嵌入式双角色架构**：nRF52840 同时当 Central（连手柄）和 Peripheral（NUS 给 GUI），`Bluefruit.begin(1,1)` 一行搞定
5. **两个手柄数据归一化**：不同格式（8字节 vs 2字节）统一到 `ParsedInput` 结构体，后续所有处理逻辑只认一种格式
