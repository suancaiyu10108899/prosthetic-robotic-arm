# LabDocs 体系维护规范 v1.0

> 最后更新：2026-07-16
> 目标：让任何新 AI 对话都能在 5 分钟内接上任务，并知道如何更新体系。

---

## 一、新对话接手指南（3 步，5 分钟）

```
步骤 1：读 CLAUDE.md → 了解全局状态（活跃项目数、本周重点、阻塞项）
         └─ 顶部 HANDOFF 块（5行）是最快的接手方式

步骤 2：如果要深入某个项目
         └─ 读该项目 CLAUDE.md → 当前阶段、设备清单、踩坑记录

步骤 3：如果项目有子系统
         └─ 读子系统 README.md → 该子系统的目标、设备、任务
```

---

## 二、对话结束时必做的事（更新检查清单）

每次 AI 协作结束时，按此清单逐项检查。不是全做——只做有变化的项。

```
□ 项目状态变了？          → 更新子项目 CLAUDE.md（当前阶段 + 已完成事项）
□ 完成了什么？            → 在子项目 日总结/ 里追加一条（日期.md）
□ 新设备打通了？          → 更新硬件资产总索引.md（新增/改状态）
□ 跨项目决策？            → 追加到 周报与汇报/决策日志.md
□ 踩坑了？                → 追加到子项目 CLAUDE.md 踩坑记录区
□ 任务状态变化？          → 更新子项目 任务看板.md

根 CLAUDE.md 更新检查：
□ 项目速览表的"下一步"是否需要更新？
□ "本周重点"是否需要更新？
□ "最近变更"是否要追加（超过 5 条时删最旧的）？
    格式：| YYYY-MM-DD | 一句话 |
□ HANDOFF 块是否需要更新？

Git 同步（必须！）：
□ git add -A && git commit -m "update: ..." && git push
    每次对话结束必须 push，确保文档变更同步到 GitHub。
    项目库和新建的代码库均为 PRIVATE，数据不公开。
```

## 三、文件更新规范

### 3.1 CLAUDE.md 更新规则

| 文件 | 更新触发 | 更新内容 |
|------|---------|------|
| 根 `CLAUDE.md` | 项目增/删/阶段变更 | 项目速览表 + HANDOFF 块 |
| 根 `CLAUDE.md` | 每次对话结束 | 最近变更 |
| 子项目 `CLAUDE.md` | 阶段推进 | 当前阶段 + 已完成事项 + 待推进 Top 5 |
| 子项目 `CLAUDE.md` | 新设备接入 | 设备速查表 |
| 子项目 `CLAUDE.md` | 踩坑 | 踩坑记录（最近 5 条） |

### 3.2 不再更新的文件

| 文件 | 触发 |
|------|------|
| `硬件资产总索引.md` | 新硬件到货 / 报废 / 项目完结 |
| `决策日志.md` | 跨项目层面的决策（不是每个技术选择） |

### 3.3 日总结格式

```markdown
# {日期} — {一句话主题}

## 做了什么
- 

## 关键决策
- （为什么这么选）

## 踩坑/教训
- 

## 下一步
- 
```

## 四、多级任务管理

### 层级定义

```
L0 — 全局       根 CLAUDE.md + README.md + 硬件资产总索引
L1 — 项目       子项目 CLAUDE.md + README.md + 任务看板
L2 — 子系统     子系统 README.md + 子系统任务看板（可选）
```

### 何时建子系统

✅ 建：该子方向有 ≥5 个独立任务，且可能独立分配给不同的 AI 对话推进
❌ 不建：只有 1-3 个任务，或高度耦合无法独立推进

### 子系统结构

```
{项目名}/
└── 子系统/
    └── {子系统名}/
        ├── README.md      ← 目标、设备清单、关键参数
        └── 任务看板.md     ← 该子系统的 TODO/DONE
```

### 多系统并行推进

当多个子系统同时工作时：
- 每个子系统在自己的目录里独立维护任务看板
- 项目层 `CLAUDE.md` 的子系统速览表显示各自的进度
- 项目层 `任务看板.md` 只放跨子系统的集成任务

## 五、HANDOFF 交接块格式

根 `CLAUDE.md` 顶部 HANDOFF 块（5 行以内）：

```markdown
> 🧭 HANDOFF（给下一个 AI）：当前在做什么、阻塞项、从哪开始。
```

子项目 `CLAUDE.md` 顶部同样格式。

每次对话结束时更新。

## 六、Git 提交规范

| 变更类型 | Commit 前缀 |
|------|------|
| 新项目启动 | `new: {项目名}` |
| 归档 | `archive: {项目名}` |
| 阶段推进 | `update: {项目名} — {做了什么}` |
| 规范/模板改进 | `improve: {改进内容}` |
| 资料归档 | `docs: {归档了什么}` |

## 七、反模式（别这么做）

- ❌ 对话结束不更新任何文件 → 下一个 AI 从零开始
- ❌ HANDOFF 块写流水账（10 行） → 精简到 5 行
- ❌ 任务看板只加不删 → DONE 超过 15 条就移到"已完结阶段"区
- ❌ 在所有层级重复写同样的信息 → L0 放全局，L1 放项目，L2 放子系统细节
- ❌ 踩坑不记录 → 下个 AI 踩同样的坑
- ❌ 决策不记录 → 三个月后不知道当时为什么这样选
