## 二、怎么做到的——五层关卡

### 第一层：扫描到手柄

`Bluefruit.begin(peripheralCount, centralCount)` 参数顺序是反直觉的——第一个是 Peripheral。改成 `begin(1,1)` 双角色模式，loop() 里每 10 秒强制重启扫描器，绕过了 Adafruit 库的扫描器休眠 bug。

### 第二层：建立连接

扫到广播名含 "LOOKBON" 的设备后 `Central.connect(report)`。用名称匹配而不是 MAC 地址硬编码——换同型号手柄自动连。

### 第三层：订阅通知（最难的）

标准流程：`discover()` → `enableNotify()` → 回调收到数据。LOOKBON 手柄的 BLE 芯片完全不响应 Adafruit 库的 GATT discover 请求。Python 参考代码用另一个库（bleak）拿到了 CCCD Handle = 7/67/131，我们在 nRF52840 端绕过 discover 直写这三个 Handle——用的是 Nordic SoftDevice 底层 API `sd_ble_gattc_write`。

但 `onConnect` 回调在 SoftDevice 中断上下文中，里面写 CCCD 然后 `delay(2000)` 会阻塞整个 BLE 栈 → 手柄 8 秒没收到心跳 → 超时断连。改成 loop() 里的阶段机：连接后每 2 秒温和写一个 CCCD，3 个写完正好 6 秒，不断连。

### 第四层：数据到达

CCCD 写完后，`onBLEEvent` 收到 `BLE_GATTC_EVT_HVX`（Handle Value Notification），数据在 `hvx->data` 里。

### 第五层：协议解码

最开始以为 LOOKBON 用 ASCII 字符串——收到的 `A3` 被当成两个字符 `'A'` `'3'`，打印出来是乱码。改成 hex 打印后看到 `0xA3`——一个字节编码了两个信息：高 4 位是事件类型（A=单击/B=长按/C=释放/D=方向），低 4 位是按键/方向编号。

## 三、踩过的最大的坑

### `while (!Serial)` 死等 — 靠自己排查发现的

MicroUSB 供电一切正常。换电池供电 → 蓝灯闪一下全灭。排查链：

1. 先怀疑供电不足 → 测电压 3.269V → 排除了
2. 换 5V 接 BAT 引脚 → 还是灭 → 不是供电问题
3. 用 MicroUSB 供电 + 不开串口监视器 → 蓝灯也不亮 → 定位到代码
4. 读 `setup()` 第一行 `while (!Serial)` — nRF52840 原生 USB 在无 Host 时 `!Serial` 永远为 true

**这个 bug AI 没发现，是在物理直觉（"蓝灯不亮→板子没启动"）驱动下自己读代码定位到的。** 改为 3 秒超时解决。

### 板① 烧毁 — 晚上对接时

与学长整合时，开发板和降压板是上下叠插结构。对接过程中可能没插稳，降压板的 8V/12V 灌进 nRF52840。测电压正常、引脚无短接（兆欧级别），但板子已无法启动。

教训：调试阶段不要叠插供电——独立 MicroUSB 供电。
